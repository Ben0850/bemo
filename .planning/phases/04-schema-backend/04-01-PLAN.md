**Kritische Detail-Regeln:**
- `CREATE TABLE IF NOT EXISTS` — idempotent, kein Migrations-Guard nötig
- `direction TEXT NOT NULL CHECK (direction IN ('in','out'))` — Inline-CHECK direkt an der Spalte (PAY-DB-03)
- `amount REAL NOT NULL CHECK (amount > 0)` — Inline-CHECK direkt an der Spalte (PAY-DB-03)
- `booked_by TEXT NOT NULL DEFAULT ''` — wird vom POST-Handler in Plan 04-02 aus `req.headers['x-user-name']` gefüllt; DEFAULT '' verhindert NULL-Constraint-Probleme bei direkter SQL-Tests
- `bank_account_id INTEGER` — KEIN NOT NULL (Bar/Kasse-Buchungen ohne Konto erlaubt). FK ohne ON DELETE CASCADE (Bankkonto-Löschen darf nicht Zahlungen mitnehmen — Verhalten wie bestehender bank_accounts-Verbund)
- `FOREIGN KEY (invoice_id) REFERENCES invoices(id) ON DELETE CASCADE` — beim Rechnung-Löschen werden alle zugehörigen Zahlungen automatisch entfernt (PAY-DB-02)
- `CREATE INDEX IF NOT EXISTS` — idempotent, ohne dieser Klausel würde der zweite Server-Start fehlschlagen
- Index-Namen sind explizit (`idx_invoice_payments_invoice_date`, `idx_invoice_payments_bank_account`) — keine SQLite-Auto-Namen verwenden, damit Tests deterministisch sind
- Position im File: NACH bank_accounts (Zeile 432), VOR `try { db.run("ALTER TABLE invoice_items ADD COLUMN vat_rate ..." } ...` (Zeile 435)

**Do NOT:**
- ALTER TABLE statements für Spalten-Migration anlegen (Tabelle ist neu, kein Migrationspfad nötig)
- Eine separate save()-Aufruf nach den db.run()-Calls einfügen (save() läuft am Ende von getDb() in Zeile 880)
- Zwischen den beiden CREATE INDEX-Statements oder davor BEGIN/COMMIT-Transaction-Blöcke einfügen
