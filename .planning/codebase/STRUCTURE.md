## Directory Purposes

**server.js:**
- Purpose: Main application server (Express.js)
- Contains: 60+ REST API endpoints organized by feature (customers, vehicles, staff, invoices, calendar, time tracking, files)
- Routes organized by domain: Customers (62-318), Vehicles (319-349), Staff (379-491), Calendar (911-965), Invoices (1118-1259), Time Tracking (2159-2498), Files (2570-2637), etc.
- Cross-cutting: Health check (47), version endpoint (16), permission guards (51-60)

**db.js:**
- Purpose: Database layer - schema definition and initialization
- Contains: 16+ table definitions (customers, vehicles, staff, invoices, time_entries, calendar_appointments, etc.)
- Migrations: Inline ALTER TABLE commands for backwards compatibility
- Helpers: `getDb()`, `queryAll()`, `queryOne()`, `execute()` for query execution

**public/:**
- Purpose: Frontend SPA (Single Page Application)
- index.html: Shell page with login screen and app container; includes nav, modals, toast notifications
- js/app.js: Complete frontend logic (state, API calls, page rendering, event handlers)
- css/style.css: Responsive design (desktop sidebar + mobile hamburger menu)

**nginx/:**
- Purpose: Reverse proxy, WAF (ModSecurity), SSL termination
- Dockerfile: Multi-stage build with ModSecurity rules
- Routes traffic to app container on internal network
- Handles HTTPS, certificate renewal via certbot

**.github/workflows/:**
- Purpose: CI/CD pipeline automation
- Triggers: On push/PR to main branch
- Tasks: Build Docker images, run tests, deploy to Hetzner server

**data/:**
- Purpose: Runtime data persistence
- bemo.db: SQLite database (created on first run, gitignored)
- Volume mount: `app-data` in docker-compose for container persistence

**Datenimport/:**
- Purpose: Data import utilities
- Scripts: fix-dates.js, import-fix.js, import.js for bulk customer/vehicle data loading
- One-time usage for initial data migration

**.planning/codebase/:**
- Purpose: GSD documentation (architecture, structure, conventions, testing, concerns)
- Generated by GSD commands (not committed, ephemeral)

## Naming Conventions

**Files:**
- Server files: camelCase.js (server.js, db.js)
- Frontend assets: kebab-case or lowercase (style.css, app.js)
- Docker: Dockerfile (no extension)
- Config: dot-prefixed (.env, .gitignore)

**Directories:**
- Lowercase plural for feature domains: `public`, `data`, `nginx`, `.github`
- Single word for clear purpose: `bilder` (images), `Datenimport` (data import)

**API Routes:**
- Kebab-case path segments: `/api/time-entries`, `/api/fleet-vehicles`
- Resource-oriented design: `/api/customers`, `/api/customers/:id`, `/api/customers/:id/vehicles`
- Nested routes for relationships: `/api/customers/:id/credits`, `/api/invoices/:id/items`
- Special actions: `/api/customers/:id/check-duplicate`, `/api/time/stamp`, `/api/staff/me/password`

**Database Naming:**
- Tables: snake_case (customers, time_entries, calendar_appointments)
- Columns: snake_case (first_name, created_at, updated_at)
- Foreign keys: `*_id` suffix (customer_id, staff_id, fleet_vehicle_id)
- Timestamps: `created_at`, `updated_at` on most tables

**Frontend Variables & Functions:**
- Global config: UPPERCASE (STATIONS, VEHICLE_TYPES, CUSTOMER_TYPES)
- State variables: camelCase (loggedInUser, currentPage, autoRefreshTimer)
- Functions: camelCase (doLogin, renderCustomers, showPage, api, showToast)
- Helper functions: camelCase prefixed (berlinNow, berlinToday, berlinTime)

**Permission Levels:**
- Values: 'Admin', 'Verwaltung', 'Buchhaltung', 'Benutzer' (German capitalized strings)
- Header: X-User-Permission, X-User-Id (PascalCase with hyphens)

## Where to Add New Code

**New REST Endpoint:**
1. Implementation: Add route handler in `server.js` near related endpoints (e.g., new customer feature → near line 62-318)
2. Database query: Add migrations/schema to `db.js` if new tables needed
3. Frontend: Add page to `public/js/app.js` or update existing page's render function
4. Authorization: Use existing permission guard pattern: `req.headers['x-user-permission']`

**New Frontend Page:**
1. Add nav link in `public/index.html` (line 44+) with `data-page="page-name"`
2. Create render function in `public/js/app.js`: `function renderPageName() { ... }`
3. Add case to `showPage()` switch statement (route dispatcher)
4. Call API via `api()` helper function
5. Update auto-refresh logic if needed (line 103-111)

**New Business Entity (Customer → Company refactor example):**
1. Add table in `db.js` with schema
2. Add CRUD routes in `server.js`
3. Add foreign keys and relationships in schema
4. Update customer page in `public/js/app.js` to render new entity
5. Add permission checks for sensitive operations

**Utilities & Helpers:**
- Shared functions: Add to appropriate section in `server.js` (e.g., time helpers at line 18-29)
- Frontend helpers: Add to `public/js/app.js` top section (validation, formatting, API wrapper)
- Database helpers: Extend `db.js` query functions or add migration patterns

## Special Directories

**data/:**
- Purpose: SQLite database persistence
- Generated: On first `getDb()` call
- Committed: No (gitignored in production)
- Mounted: `app-data` volume in Docker for persistence across container restarts

**nginx/**
- Purpose: Reverse proxy, WAF, SSL/TLS termination
- Generated: No (source-controlled)
- Committed: Yes (includes nginx.conf)
- Modified: By certbot for SSL certificates

**.github/workflows/:**
- Purpose: CI/CD automation
- Generated: No (source-controlled)
- Committed: Yes (workflow files are part of repo)
- Triggered: On push to main branch, creates Docker images, deploys to server

**.planning/codebase/:**
- Purpose: GSD documentation (ephemeral analysis documents)
- Generated: Yes (by GSD commands)
- Committed: No (local only)
- Updated: When running `/gsd:map-codebase` commands

- Used: Frontend branding, email templates (implicit)
